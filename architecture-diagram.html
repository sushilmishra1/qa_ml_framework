5. Design rationale

  • Layered, not monolithic: ingestion / features / model / decision / reporting are separate modules — each independently testable and swappable (e.g. Locust parser could be replaced by JMeter without touching the model layer).
  • Time-split CV, not k-fold: test results are time-ordered; k-fold would leak future data into training and produce offline metrics the model can't actually hit in production.
  • Random Forest over XGBoost: feature_importances_ gives an interpretable answer to "why was this test flagged" when the gate blocks a PR.
  • Unsupervised where labels don't exist: flaky-test detection and perf anomalies have no ground-truth labels, so Isolation Forest is used instead of forcing a supervised approach.
  • The gate is a pure function of config: risk_threshold and top_n_tests live in config.yaml, tunable per team/repo without a code change.
  • CI is the actual product surface: the ML output is wired directly into the PR merge decision, not just a dashboard nobody looks at.

6. Known limitations (not yet solved)

  • Validated on synthetic data only. Never run against a real project's accumulated CI/perf history — only against the synthetic generator's cleaner, hand-injected patterns.
  • No drift monitoring. Nothing tracks whether a deployed model's accuracy degrades as the suite and codebase evolve.
  • Gate averages top-N risk. One severely at-risk test can be diluted by nine lower-risk ones and still pass the mean-based threshold.
  • Rolling z-score features self-normalize. A regression that outlasts baseline_window samples gets absorbed into its own rolling baseline and the anomaly score decays back toward zero.
